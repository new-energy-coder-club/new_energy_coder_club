# HXC_A 开发使用说明

## 1. 目的

这份说明基于只读梳理以下目录后整理而成：

- `gripper_debug`
- `HCX_A_N630`
- `HXC_A_C620`
- `AM32`
- `BM-M1505B`

目标不是复述每个文件，而是提炼 `HXC_A` 这套板卡/工程模板的稳定用法，方便后续继续开发。

## 2. 结论先看

当前目录里的 `HXC_A`，本质上是一套基于 `ESP32-S3` 的电机控制板模板，至少已经被复用了三次：

- `HXC_A_C620`：单 `C620` 调试工程
- `HCX_A_N630`：单 `N630/VESC` 调试工程
- `gripper_debug`：`C620 + N630/VESC` 混合总线集成调试工程

仓库里出现的 `HCX_A_*` 路径按当前口径视为命名错误，不代表另一块板。

`BM-M1505B` 虽然不是 `HXC_A` 命名，但它沿用了同一类 `ESP32 PlatformIO temple project` 模板，说明这套模板是通用板级开发骨架，不是只为某一台电机写死的。

## 3. 板级共性

### 3.1 MCU / 工程骨架

- 板卡类型：`ESP32-S3-DevKitC-1`
- 开发框架：`Arduino + PlatformIO`
- 常见 Flash / PSRAM 配置：
  - `board_build.flash_size = 16MB`
  - `board_build.partitions = default_16MB.csv`
  - `board_build.arduino.memory_type = qio_opi`
- 常见环境名：`temple_project`

### 3.2 串口约定

- 串口波特率统一按 `115200`
- `GPIO44` = UART0 RX
- `GPIO43` = UART0 TX
- 工程里普遍开启了 `ARDUINO_USB_CDC_ON_BOOT=1`

这意味着：

- `Serial` 更偏向 USB CDC
- `Serial0` 才是连到外部无线烧录器/串口链路的 UART0

所以后续如果你写需要走外部串口链路的交互代码，优先延续这套初始化：

```cpp
Serial0.begin(115200, SERIAL_8N1, 44, 43);
```

`gripper_debug` 比单电机工程更稳一点，它把日志同时输出到 `Serial` 和 `Serial0`，这对 USB 调试和无线串口并存很有价值，后续可以继续沿用。

### 3.3 CAN 约定

- `GPIO8` = CAN TX
- `GPIO18` = CAN RX
- 用的是 ESP32 自带 `TWAI`
- 工程里按“经典 CAN”写，不是 CAN FD

绝大多数代码直接写死为：

```cpp
twai_timing_config_t t_config = TWAI_TIMING_CONFIG_1MBITS();
```

但对 `N630/VESC`，这里不应该把波特率写成“固定知识”。更合理的开发规则是：

- 调试前先接 `VESC` 上位机看当前实际波特率
- 再把 ESP32 侧 `TWAI` 配置对齐到当前硬件

也就是说，`N630` 波特率应视为“每次调试前都能快速确认的运行时前置检查”，而不是要替操作者永久记住的静态知识。

## 4. 三类工程分别怎么用

### 4.1 `HXC_A_C620`：单 C620 调试模板

适合做：

- 单 `C620` 起转
- 电流环/速度环基础调试
- 电机 ID 排查
- CAN 接线排错

这个工程的特点：

- 标准 `11-bit` CAN 帧
- 控制 ID：
  - 电机 `1~4` 用 `0x200`
  - 电机 `5~8` 用 `0x1FF`
- 反馈 ID：
  - `0x200 + motor_id`
- 控制本质仍然是发电流给定
- “速度模式”是在 ESP32 侧用反馈转速做了一个很简单的外环

串口命令风格偏短命令：

```text
r 3000
i 3.5
l 8
s
scan
diag
```

建议把它理解成：

- `r`：速度模式
- `i`：电流模式
- `l`：限流
- `s`：停机
- `scan`：主动扫描可能的电机 ID
- `diag`：抓一份诊断快照

这个工程还做了两件很有价值的事：

1. 上电先用 `listen-only` 模式只听总线，不先乱发控制帧。
2. 后续可以主动发送“零电流扫描帧”来找电机 ID。

如果后面你要继续做 `C620` 方向开发，优先从这个工程演化，而不是从零重写。

### 4.2 `HCX_A_N630`：单 N630 / VESC 调试模板

适合做：

- 单 `N630` 起转
- 纯 RPM 指令链路打通
- VESC CAN 报文格式确认

这个工程的特点：

- 用扩展帧，不是标准帧
- 发送 `SET_RPM`
- 周期性重复发送目标值，周期大约 `20 ms`
- 串口直接输入一个数字就会更新目标转速

典型输入：

```text
3000
-1500
0
```

但这里要注意两点：

1. 调试前先在 `VESC` 上位机里确认当前 CAN 波特率。
2. 在 `gripper_debug` 的集成工程里，作者已经明确把这个值当作“更像 ERPM 的指令路径”，而不是保证等于机械 RPM。

所以后续如果你做 `N630` 控制闭环，不要直接把这个输入值包装成“机械转速事实”。

### 4.3 `gripper_debug`：C620 + N630 混合集成模板

如果你的后续目标是夹爪、机构联动、混合总线调试，最值得沿用的是这个工程。

它已经把两种协议放到同一套安全框架里：

- `C620` 走标准帧
- `VESC/N630` 走扩展帧
- 两者共用一套 TWAI 控制器

它相对前两个单电机工程最大的价值，不是“功能更多”，而是“安全边界更明确”：

- 上电后默认不输出
- 有 boot lockout
- 必须先 `arm`
- 有串口活动 watchdog
- 超时后会停机
- `C620` 在位置模式超时时会进入低电流保持，而不是直接继续跑

典型命令：

```text
arm
disarm
keepalive
status
cfg show
c rpm 2000
c current 3.5
c zero
c pos 180
v rpm 1 3000
v stop
stop all
```

其中几个关键点：

- `c pos` 不是 `C620` 原生位置模式，而是 ESP32 自己拿角度反馈做的软件位置环
- `c zero` 会把当前机械位置设成软件零点
- `v pos` 在这个工程里已经被明确禁用，因为实机验证出过危险跑飞

如果后续你要做：

- 位置控制
- 多电机联动
- 带安全保护的调试固件
- 夹爪/执行器类集成逻辑

建议直接基于 `gripper_debug` 扩展。

## 5. 无线烧录 / 上传方式

按当前口径，后续应以 `cmsis-dap` 作为目标烧录/调试链路，而不是把 UART `esptool` 方案当成标准答案。

更准确的说法应该是：

- 想走的是“两只 DAP 烧录器之间通信”的 `cmsis-dap` 路线
- 仓库里早期文档还有一些偏向 UART `esptool` 的说明
- 这条 `cmsis-dap` 路线目前并不成熟，实际跑起来比较难搞，也不是常用稳定流

所以后续写说明或改工程时，建议口径写成：

- 标准开发交互仍然围绕 `Serial0 + CAN`
- 无线/远程烧录的目标方案是 `cmsis-dap`
- 但这部分当前属于“在尝试、未完全跑顺”的能力，不要在文档里伪装成已经稳定

## 6. 从 AM32 / BM-M1505B 能学到什么

这两个目录不是 `HXC_A` 本体，但能帮助理解这套开发风格：

- `AM32` 说明这套仓库习惯把“电机协议适配”和“板级模板”分离
- `BM-M1505B` 给出了一个更完整的“查询优先、模式切换、反馈解析、命令壳层”示例

特别是 `BM-M1505B` 的价值在于，它证明了这块 ESP32-S3 模板不仅能做“单命令起转”，也适合做：

- 查询型协议
- 模式切换
- 主动上报 / 查询上报双模式
- 更系统的串口调试壳层

因此如果你后续要把 `HXC_A` 做成更通用的电机调试母板，`BM-M1505B` 的工程组织方式值得参考。

## 7. 推荐的后续开发入口

按目标选入口，不要混着改：

### 目标 A：只调 `C620`

从 `HXC_A_C620/.../ESP32_platformio_temple_project` 开始。

适合继续加：

- 更稳的速度环
- 位置环
- 电机识别/诊断
- 参数保存

### 目标 B：只调 `N630/VESC`

从 `HCX_A_N630/ESP32_platformio_temple_project` 开始。

适合继续加：

- 更明确的状态反馈解析
- 真正可验证的速度单位定义
- 安全停机与限幅

### 目标 C：做机构集成、夹爪、双电机联调

从 `gripper_debug` 开始。

这是目前仓库里最像“后续正式开发基线”的工程。

## 8. 以后改代码时最好守住的规则

- 不要随手改 `GPIO8/18` 和 `GPIO43/44`，除非硬件已经确认变更。
- 不要把 `N630` 波特率写成固定知识。
- 调试 `N630` 前，先接 `VESC` 上位机确认当前配置。
- 不要把 `VESC` 扩展帧逻辑和 `C620` 标准帧逻辑混到一个解析分支里。
- 不要把 `gripper_debug` 里的安全门槛删掉再试电机。
- 不要把 `v pos` 这种未验证路径重新打开，除非你先把协议和量纲验证清楚。
- 如果只是做机构层逻辑，优先复用现有命令壳层，而不是再造一套新的串口命令语法。

## 9. 当前口径

- 正式命名统一用 `HXC_A`
- `N630` 波特率不写死，调试前接 `VESC` 上位机实时确认
- 目标烧录链路按 `cmsis-dap` 口径，但当前实现并不成熟
