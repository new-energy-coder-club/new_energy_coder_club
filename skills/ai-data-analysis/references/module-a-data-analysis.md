# 模块 A：智能数据分析（固定同步接口版）

> 本文档是 `ai-data-analysis` Skill 数据分析业务流程的详细规范。
> 主 Skill 入口：`SKILL.md`。

## 核心背景

智能数据分析基于 **自然语言**：

- 本 Skill 通过**固定的同步 HTTP 接口**直接调用，服务端等待上游 ChatBI 完整响应后返回 Markdown 结论。
- 正常耗时 **20~90 秒**，偶发 **3 分钟**；脚本超时 `260s`，服务端内部 `240s` 超时。
- **不再**依赖"算子发现 / 算子详情 / 算子调用"的通用路由流程。

## 目标接口


| 字段         | 值                                                                       |
| ------------ | ------------------------------------------------------------------------ |
| URL          | `https://y.tencentmusic.com/openapi/v1/musician/data/agent/chat-sync` |
| Method       | `POST`                                                                   |
| Content-Type | `application/json`                                                       |
| 认证 Header  | `tme-header-token: <TOKEN>`                                              |
| 调用方式     | 同步（SYNC），一次 POST 直接拿到完整 Markdown 结论                       |
| 调用脚本     | `scripts/chat_sync.py`                                                   |

### 请求入参


| 字段          | 类型   | 必填 | 说明                                                                                      |
| ------------- | ------ | ---- | ----------------------------------------------------------------------------------------- |
| `userMessage` | string | 是   | 用户的原始自然语言提问，**不要做任何加工/改写**。字段名严格为 `userMessage`               |
| `songId`      | number | 否   | 仅当用户明确指向单首歌曲、**且已按「songId 获取流程」通过整体分析 content 精准匹配到该歌名对应的 songId** 时传入；整体数据/多首歌概览**不要传**；**严禁**凭印象、模糊匹配或服务端猜测传入 |

> ⚠️ **上表即完整请求参数，不存在第三个合法字段。**

#### 🚫 常见错误字段黑名单（服务端遇到会返回 `code=100010`）

| 错误字段         | 说明                                                                                       |
| ---------------- | ------------------------------------------------------------------------------------------ |
| `query`          | ❌ 这是其他通用问答接口的字段名，本接口的字段是 `userMessage`，绝不要混用                  |
| `question` / `prompt` / `message` / `text` / `input` / `content` | ❌ 不要凭"感觉"发明字段名                           |
| `sessionId`      | ❌ **这是响应字段**（`data.sessionId`），由服务端生成返给你，**请求里传它一定会失败**       |
| `accountId` / `userId` / `uid` | ❌ 网关会根据 `tme-header-token` 自动解析当前登录用户，**调用方传就会失败** |
| `excludeFraudulentPlays` / 新旧口径参数 | ❌ 服务端根据账号灰度自动判定                                        |

#### ❌ 错误调用反例（**严禁模仿**）

```json
// ❌ 错误 1：字段名错误 + 传了响应字段 sessionId
{
  "query": "分析《文艺复兴》这首歌近期的播放量表现",
  "sessionId": "chat-bi.data.agent:chat-bi.data.agent:835586:20260422"
}
// 服务端返回：{"code":"100010","message":"数据异常，请稍候再试",...}

// ❌ 错误 2：自己 curl 直连且改字段名
// curl -X POST ".../chat-sync" -d '{"question":"...", "accountId":123}'
```

#### ✅ 唯一正确的调用方式

```bash
# 整体数据分析
python3 scripts/chat_sync.py "分析我最近 30 天的播放量表现"

# 指定歌曲分析
python3 scripts/chat_sync.py "分析《文艺复兴》这首歌近期的播放量表现" 987654
```

> 🔐 **accountId 无需传入**：服务端会通过请求 header 中的 `tme-header-token` 由网关解析出当前登录用户的 accountId，调用方**不应**在 body 里传 `accountId`。
>
> 🚫 **sessionId 不要传**：这是响应字段而非请求字段，传入必报错。
>
> 🚫 **不要**传 `excludeFraudulentPlays`、新旧口径等任何其他参数 —— 服务端根据账号灰度自动判定。

## 业务流程

### Step 1：意图识别与参数收集

从用户输入中提取两个核心要素：

1. **`userMessage`**（必填）：用户的自然语言数据问题。通常就是用户的原始提问，**不要做任何加工重写**。例如：

   - 「我最近7天的播放量怎么样？」 → `userMessage = "我最近7天的播放量怎么样？"`
   - 「帮我看看我的听众画像」 → `userMessage = "帮我看看我的听众画像"`
   - 「分析这首歌的表现」（上下文已有 songId） → `userMessage = "分析这首歌的表现"`
2. **`songId`**（可选）：当用户明确提到某一首具体歌曲时传入，**必须严格遵守下文「songId 获取流程」三档判定**：

   - ✅ **唯一合法来源**：同会话前一轮（或本轮主动发起）的**整体数据分析 content**，其中含 `songId ↔ songName` 对照信息
   - ✅ **档 A：精准匹配**：用户所指歌名与 content 中某条 `songName` 完全相等且仅命中一条 → 直接取出对应 songId 传入
   - ❔ **档 B：相似命中**：存在包含/被包含/大小写标点差异/同名多条等明显相似条目时 → **输出「候选确认话术」让用户选择**；未得用户确认前不传 songId、不调单曲分析
   - ❌ **档 C：完全无关 / content 中没有单曲信息 / 歌名缺失** → 按「匹配失败兜底话术」输出，**不传 songId、不退化为让服务端猜、不追问技术性信息**
   - ❌ **严禁**编造 songId、从不相关上下文截取数字当 songId、基于“大概像”自行判定

> 🚨 **铁律**：`songId` 必须与用户所说那首歌**一一对应**。相似还是不相似的拽板 **交给用户**，不是 AI 自己判“就是它了”；AI 拽板错 = 服务端按错的歌返回 = 用户看到错结果。

> 🔐 `accountId` 无需手动收集或传入，服务端会根据 `tme-header-token` 由网关解析出当前用户。

### Step 2：确认登录态

确保环境变量 `TME_HEADER_TOKEN` 已设置且有效，详见主 `SKILL.md`「第一步：确保登录态」。

未登录时禁止直接调用本接口。

### Step 3：调用 `chat_sync.py` 脚本

> ⚠️ **强制约束**：**必须**通过 `scripts/chat_sync.py` 调用接口，**严禁**自行使用 HTTP 工具、`curl`、`requests`、`urllib`、`fetch` 等方式直连接口。脚本里已写死正确的 URL、Header、字段名（`userMessage` / `songId`）、超时时间，任何绕过 = **必然出错**（最典型表现就是服务端返回 `code=100010` "数据异常，请稍候再试"）。

```bash
python3 scripts/chat_sync.py "<userMessage>" [songId]
```

示例：

```bash
# 最小入参
python3 scripts/chat_sync.py "我最近7天的播放量怎么样？"

# 指定歌曲
python3 scripts/chat_sync.py "分析这首歌最近的表现" 987654
```

脚本会：

- 从环境变量 `TME_HEADER_TOKEN` 读取登录态
- 以 `Content-Type: application/json` + `tme-header-token: <TOKEN>` 向固定 URL 发起 POST
- 将接口返回的原始 JSON 格式化后打印到 stdout

> 网关会根据 `tme-header-token` 解析出当前登录用户的 accountId，并注入到请求中传给后端，所以脚本无需传 accountId。

等价的原始 HTTP 请求（**仅供阅读理解底层协议，禁止实际执行！** 请严格使用 `scripts/chat_sync.py`）：

```bash
# ⚠️ 下面这段 curl 仅是对脚本内部行为的说明，不要让 Agent 亲自去执行
curl -X POST \
  "https://y.tencentmusic.com/openapi/v1/musician/data/agent/chat-sync" \
  -H "Content-Type: application/json" \
  -H "tme-header-token: $TME_HEADER_TOKEN" \
  --max-time 260 \
  -d '{
    "userMessage": "我最近7天的播放量怎么样？"
  }'
```

注意：

- body 有且只有 `userMessage`（必填）、`songId`（可选）两个字段
- **禁止**出现 `query`、`sessionId`、`accountId`、`question`、`prompt` 等字段名
- `sessionId` 是响应字段而非请求字段

### Step 4：解析返回

接口返回标准 `BaseResponse`，`code == 0` 表示成功：

```json
{
  "code": 0,
  "message": null,
  "data": {
    "sessionId": "chat-bi.data.agent:chat-bi.data.agent:<accountId>:20260422",
    "messageId": "ag-msg-xxxx",
    "content": "...Markdown 分析结论...",
    "finishReason": "COMPLETE",
    "safetyRevoked": false
  }
}
```

> `sessionId` 中的 `<accountId>` 由服务端根据 `tme-header-token` 解析后填充，调用方无需关心。

读取关键字段：

```python
payload = json.loads(stdout_of_chat_sync)
if payload["code"] != 0:
    # 走「业务错误处理」逻辑
    ...
else:
    data = payload["data"]
    content = data["content"]             # Markdown 文本
    finish_reason = data["finishReason"]
    safety_revoked = data["safetyRevoked"]
```

### Step 5：根据 `finishReason` 处理并展示


| finishReason     | 含义                  | 处理                                                                                   |
| ---------------- | --------------------- | -------------------------------------------------------------------------------------- |
| `COMPLETE`       | 正常完成              | **完整原样**把 `data.content` 展示给用户（见下方展示规范）                             |
| `SAFETY_REVOKED` | 输入/输出命中安全校验 | 提示用户："您的问题或 AI 回答中可能包含敏感内容，请调整提问方式后重试。"；**不要**重试 |
| `TIMEOUT`        | 同步等待超时          | 提示用户："数据分析耗时较长，请稍后再试~"；**不要**直接重试，接口不幂等                |
| `UPSTREAM_ERROR` | 上游 ChatBI 异常      | 可重试**1 次**；仍失败则提示用户"服务繁忙，请稍后再试"                                 |

## songId 获取流程与兜底话术（**单首歌分析前必读**）

> 🚨 本节与主 `SKILL.md` 的「第 1.5 步：songId 获取铁律」是同一套规范，这里给出执行细节与判定边界。**与本节冲突的其他表述一律以本节为准。**

### 触发条件

用户的本次提问**明确指向单首歌曲**，例如：

- 「分析《XXX》这首歌」
- 「《XXX》表现怎么样」
- 「这首歌最近播放量如何」（有上下文可定位到某一首歌）
- 「《XXX》还要不要继续推」

若用户问的是账号整体/多首歌概览（如「我最近数据怎么样」），本节**不适用**，直接按不传 songId 的整体分析流程执行。

### 执行步骤

#### Step ①：检查上下文是否已有"整体分析 content"

- ✅ 同一会话前面已成功执行过一次整体数据分析（用户先问了「我整体数据怎么样」或本 Agent 先前主动发起过一次） → 直接复用该 content，跳到 Step ③
- ❌ 没有 → 进入 Step ②

#### Step ②：内部发起一次整体数据分析（仅用于提取 songId 映射）

```bash
python3 scripts/chat_sync.py "<能引导服务端返回歌曲列表的提问>"
```

`userMessage` 的选取策略：

- 若用户原问题本身已经隐含账号整体含义，可直接使用用户原问题
- 若用户原问题是纯粹的单曲追问（只围绕某一首），使用固定兜底 userMessage：`"看看我最近的整体歌曲表现"`（不得改写得过于生僻，以免服务端返回无单曲信息的画像类结论）

⚠️ 本次调用属于**内部数据收集**，其返回 content **不直接展示**给用户（除非同时用户本轮也确实想看整体），否则会让用户误以为 Agent "答非所问"。

#### Step ③：从 content 中提取 (songId, songName) 映射

扫描 content 中所有可能承载单曲信息的区块，包括但不限于：

- 「热门歌曲 / 歌曲 Top / 单曲播放量排行 / 歌曲排行榜」等 Markdown 表格
- 「代表作《XXX》」「主打歌《XXX》」等标注块
- 分析主角标题如「《XXX》近期表现」

对每条记录提取 `(songId, songName)`，构成本次可用的候选映射集合 `M`。

#### Step ④：匹配结果三档判定

将用户提到的歌名记为 `Q`（去除外层书名号 `《》`、前后空白后的规范化形式）。**严格按三档判定，不得跨档降级或升级**：

| 档位 | 判定条件                                                                                     | 处理行为                                                              |
| ---- | -------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| A    | `M` 中**恰有一条** `songName` 规范化后与 `Q` **完全字符相等**                               | ✅ 取该条 songId，直接进入单曲分析，**无需询问用户**                      |
| B    | `Q` 与 `M` 中一条或多条 songName **明显相似**（包含/被包含、大小写全半角差、同名多条等） | ❔ **输出候选确认话术**，让用户从候选中选一；**不直接调 chat_sync、也不走兜底**   |
| C    | `Q` 与 `M` 内**任何**条目都无明显相似 / `M` 为空 / `Q` 为空                                | ❌ 输出「匹配失败固定兜底话术」，**不传 songId**                              |

档 B 的典型相似信号（**任满足其一**）：

- **包含 / 被包含**：`Q="文艺复兴"` 与 `songName="文艺复兴(Remix)"`
- **全半角 / 大小写 / 标点差异**：`Q="hello"` 与 `songName="Hello"`；`Q="夜曲！"` 与 `songName="夜曲"`
- **副标题不同**：`Q="XXX"` 与 `songName="XXX(新版)"`
- **同名多条**：`M` 中多于一条与 `Q` 完全相等（同名异曲）

> **规范化规则**：只去除外层 `《》` 与首尾空白；**不做**同义/近义改写、**不做**音近字替换。相似与否的最终拽板**必须交给用户**，而不是 AI 自己判定“就是它了”。

> 一律按未命中处理的圣虚描述已不适用：**相似则问，无关则兜底**。

#### Step ⑤：得到确定的 songId 后发起单曲分析（档 A 适用）

```bash
python3 scripts/chat_sync.py "<userMessage 原样>" <songId>
```

- `userMessage` **保持用户原问题原样**，不要重写
- `songId` 为 Step ④ 取出的数字

随后**原样展示** content；由于**已传 songId**，**不要**追加"末尾引导语"。

### 相似命中时的「候选确认话术」（档 B 专用，**严格遵守模板**）

触发档 B 时，**直接向用户输出下述话术**，不要先调 `chat_sync.py` 做单曲分析：

```
没有找到与《{USER_SONG_NAME}》完全一样的歌曲，您是想分析以下哪一首呢？

1. 《{candidate_1}》
2. 《{candidate_2}》
3. 《{candidate_3}》

请回复歌曲名或前面的编号；如果都不是，也可以告诉我“都不是”。
```

**填充规则**：

- `{USER_SONG_NAME}`：用户原始歌名（保留原有书名号/原文写法，不改写）
- `{candidate_n}`：从相似候选中拿 `songName` **原样值**（保留副标题、Remix 后缀等），外层用中文书名号 `《》` 包裹；若 songName 自带 `《》` 则不再重复加
- 数量：**最少 1 条、最多 5 条**；超过则按优先级（完全相等 > 互相包含 > 仅大小写/标点差异）截取前 5 条
- songId 相同的候选合并为一条，不重复列出
- 候选为空时 → 回退到档 C（兜底）

**措辞与格式铁律**：

- ❌ **绝对禁止**暴露技术字眼：`songId`、「映射」「候选池」「规范化」「相似度」「模糊匹配」「Top N」
- ❌ 绝对禁止在同一条回复里同时输出「候选确认话术」与「兜底话术」
- ❌ 绝对禁止在未获得用户确认前惄自调 `chat_sync.py`（即使你感觉某一首「最像」）
- ❌ 绝对禁止附加数据性提示（如「这几首中某歌播放量最高」）暗示用户选哪一条 —— 选择权完全交给用户

**用户回复后的处理**：

| 用户回复                           | 处理                                                                                       |
| ------------------------------ | ------------------------------------------------------------------------------------------ |
| 回复编号（如「1」「第二个」「二」）   | 取对应序号的候选 → 使用其 songId 发起单曲分析                                         |
| 回复歌曲名（如「文艺复兴(Remix)」）| 在候选集合内再次执行档 A/B/C 判定；命中则取 songId，仍不命中则走档 C                   |
| 回复「都不是」/「没有」/否定回答      | 走档 C，输出兜底话术                                                                 |
| 给出全新歌名（不在候选中）            | 当作新一轮提问，重新走 Step ①～④                                                      |
| 用户只用模糊词（如「中间那首」）     | 按模糊词在候选内再匹配；如仍精准不出，继续追问一次请用户回复编号；再不明确 → 走档 C |

### 匹配失败的固定兜底话术（档 C 专用，严格遵守措辞）

```
很抱歉，暂时不支持对《{USER_SONG_NAME}》这首歌进行数据分析，您可以尝试分析其他歌曲~
```

- `{USER_SONG_NAME}` 替换为用户提问中出现的**原始歌名**（保留书名号或原文写法），不做任何改写
- 若用户未给出明确歌名（如只说「这首歌」「那首」），整段 `《{USER_SONG_NAME}》` 替换为「该歌曲」
- **严禁**暴露以下任何技术性表达：`songId`、"映射"、"未识别到"、"未找到匹配"、"请提供更多信息"、"请确认歌曲全名"
- **严禁**在此回复末尾追加"整体数据分析末尾引导语"（那是整体分析独有）
- 允许在兜底文案后柔性附加一句引导（**非强制**）：

  ```
  您也可以问问整体歌曲表现、听众画像等整体维度的数据哦~
  ```

### 铁律汇总（必读）

1. ✅ songId 必须**来源于当前账号**的整体分析 content
2. ✅ 档 A（完全字符相等且唯一） → 直接用；档 B（明显相似） → **询问用户确认**；档 C（无关 / 空） → 固定兜底话术
3. ❌ 严禁编造 songId、从无关上下文抓数字、用旧 songId 套新歌名
4. ❌ 严禁在用户已明确指定单曲的语境下退化为“不传 songId 让服务端猜”（服务端猜错用户感知即为错结果）
5. ❌ 严禁向用户追问 `songId` / “完整歌名” / “演唱者” 等技术性补充信息（候选确认是唯一允许的用户交互形式，且必须用话术模板）
6. ❌ 严禁把候选确认话术、兜底话术、整体分析末尾引导语三者混在一条回复里输出

## 业务错误处理（`code != 0`）

| 场景                | message / code 关键字            | 处理                                                                      |
| ------------------- | -------------------------------- | ------------------------------------------------------------------------- |
| 未登录 / Token 失效 | 非 0 code，提示未登录 / 未知用户 | 重新执行主 SKILL 的登录流程获取 Token；获取后必须用新 Token 替换旧值      |
| 日配额用尽          | 「使用次数达 10 次」             | **不要重试**，直接告知："您今天的数据助手提问次数已达上限，欢迎明天再来~" |
| 歌曲命中安全打击    | 「歌曲不支持分析」               | 告知用户："很抱歉，这首歌当前不支持数据分析。换一首试试？"；**不要**重试  |
| 参数错误            | 「userMessage 不能为空」         | 修正入参后重试                                                            |
| **请求字段非法 / 绕过脚本调用** | `code=100010` / 「数据异常，请稍候再试」 | **不要盲目重试**。极大概率是因为绕过了 `scripts/chat_sync.py`，自己拼了带 `query`/`sessionId`/`accountId` 等非法字段的请求。必须改用 `python3 scripts/chat_sync.py "<userMessage>" [songId]` 正确调用 |

## 展示规范

> **核心原则：你是"搬运工"，不是"编辑"。** 接口返回的 `data.content` 已经是后端 ChatBI 生成的 **完整分析结论**，包含多级标题、表格、段落解读、建议等结构化内容。你的唯一职责是将其**原封不动**地交给用户，在此基础上仅允许按规则追加一句引导语。

### ✅ 必须做

1. **100% 原样展示 content 全部内容**：逐字符复制粘贴，字符数只能 ≥ 原文（允许多一句导语 + 末尾一句引导语），**不得少于原文**
2. **保留所有 Markdown 结构**：所有 `##`、`###` 标题、表格、代码块、图片、链接、列表、引用块、加粗、Emoji 等原样保留，不得合并/拆分/扁平化
3. **保留所有章节**：例如"一、整体播放表现"、"二、平台差异化表现"、"三、互动数据表现"、"四、趋势判断与建议"、"五、数据补充说明"等章节，**必须一个不少全部输出**，即使某章节你主观认为"不重要"也不得省略
4. **链接展示完整**：所有 URL（尤其是带签名参数的 CDN 链接）必须展示完整 URL，禁止截断或用 `...` 代替
5. **数字精确一致**：所有数字、百分比、同比环比、日期区间、占比、排名必须与原文一字不差
6. 展示前可加**一句**简短导语（如「📊 这是您的数据分析结果：」）；除此之外正文中间**不得**插入任何自己的内容

### ❌ 禁止做（**这是本 Skill 最常见的错误类型，违反将严重损害用户体验**）

1. ❌ **严禁"要点化精简"**：不得将多段落、多章节的详细分析结论改写为几个 bullet point 或简短概括
   - 反例：content 里有 5 个大章节、几十行文字，你输出时却压缩成 4 条要点、每条一句话 —— **严重违规**
2. ❌ **严禁"重排版"**：不得把段落改成表格，也不得把表格改成段落；不得调整章节顺序；不得合并相邻章节
3. ❌ **严禁"同义改写"**：不得用自己的话重新表达原文，哪怕你觉得"更简洁"或"更通顺"
4. ❌ **严禁裁剪任何细节**：包括但不限于平台对比解读、收藏/评论分析、趋势判断建议条目、数据补充说明、元数据口径说明等
5. ❌ **严禁用"..."、"等"、"（略）"** 等省略性表达代替任何一段内容
6. ❌ 严禁自行裁剪表格行数（例如"以下为前 5 行"）
7. ❌ 严禁改写数字、百分比、日期等具体数据
8. ❌ 严禁将 Markdown 转为纯文本
9. ❌ 严禁出现 "Suno" / "suno" / "SUNO" 任何形式字样（如 content 中包含，过滤后再展示）
10. ❌ 严禁对结论做脱离数据的推断或额外补充

### 🔍 输出前自检清单（强制）

在把回复发给用户之前，请逐条核对：

- [ ] 我输出的内容字符数是否 ≥ 原始 content 的字符数？
- [ ] 原始 content 里每一个 `##`、`###` 标题是否都原样出现？
- [ ] 原始 content 里每一个表格是否都完整保留（行数、列数、单元格内容一致）？
- [ ] 原始 content 里每一个数字、百分比是否都一字不差？
- [ ] 是否**没有**用自己的话重新组织、概括、改写任何段落？
- [ ] 如果是整体数据分析（未传 songId），是否按规则在末尾追加了引导语？
- [ ] 如果是单首歌分析（已传 songId），是否**没有**追加那句引导语？

**任何一条不通过，都必须重新输出完整 content。**

## 整体数据分析的"末尾引导语"（强制追加）

### 触发条件

本次 `chat_sync.py` 调用 **未传入 `songId`**，即属于"整体数据分析"场景：

- 用户问："我的数据怎么样"、"最近播放量如何"、"看看我的整体表现"、"听众画像"、"粉丝分布"、"地域分布"、"整体歌曲表现"等
- 本质特征：分析对象是**账号整体 / 多首歌概览**，而不是某一首具体的歌

### 追加规则

1. **位置**：紧跟在 content 全文之后，用一个空行隔开，作为整条回复的**最后一行**
2. **格式**（严格遵守模板，不得改写措辞）：

   ```
   💡 如果需要进一步分析单首歌曲的表现，可以告诉我歌曲名，例如：分析歌曲《{TOP_SONG_NAME}》的表现
   ```
3. **`{TOP_SONG_NAME}` 取值逻辑**（按优先级逐条尝试）：
   1. content 中"热门歌曲 / 歌曲 Top / 单曲播放量排行 / 歌曲排行榜"等表格/列表里**播放量最大 / 排名第 1** 的歌曲名
   2. content 中被显式称为"代表作《XXX》"、"主打歌《XXX》"的歌曲名
   3. content 标题或开头提到的、作为本次分析主角的那一首歌曲名（例如标题《文艺复兴》播放量分析）
4. **歌曲名格式**：
   - 外层使用中文书名号 `《》` 包裹
   - 如果原文里歌名本身已带 `《》`，则沿用原文，**不要重复包裹**
5. **兜底策略**：若 content 中**确实无法**识别任何具体歌曲名（例如只有总播放量、听众年龄分布、地域分布等，完全没有出现单曲信息），退化为通用文案：

   ```
   💡 如果需要进一步分析单首歌曲的表现，可以告诉我歌曲名，例如：分析歌曲《歌曲名》的表现
   ```

### 严禁做的事

- ❌ 当本次调用**已传入 `songId`**（单首歌分析）时，**不要**追加这段引导 —— 会造成用户困扰
- ❌ 把引导语放在 content 中间或开头（必须是整条回复的**最后一行**）
- ❌ 修改引导语的文案、表情或格式（必须严格使用上方模板）
- ❌ 编造 content 中并未出现的歌曲名来填入 `{TOP_SONG_NAME}`

## 场景示例

### 场景一：泛数据分析（整体数据，未传 songId）

**用户**：「我的数据怎么样？」

**Agent 行为**：

1. 确保已登录（`TME_HEADER_TOKEN` 有效）
2. 调用（**不传** `songId`）：
   ```bash
   python3 scripts/chat_sync.py "我的数据怎么样？"
   ```
3. **完整原样**展示返回的 Markdown（所有章节、表格、段落、数字都不得精简或改写）
4. 由于**未传 `songId`**，属于整体数据分析 → 在 content 全文之后、回复的**最末尾**追加一行引导语：
   ```
   💡 如果需要进一步分析单首歌曲的表现，可以告诉我歌曲名，例如：分析歌曲《{TOP_SONG_NAME}》的表现
   ```
   其中 `{TOP_SONG_NAME}` 替换为 content 中识别到的**播放量最大**的那首歌名（如《文艺复兴》），外层用 `《》` 包裹

### 场景二：指定歌曲分析（已传 songId）

**用户**：「帮我看看《XXXX》这首歌最近表现怎么样」

**Agent 行为**：

1. **先走「songId 获取流程」**：
   - 检查会话上下文是否已有整体分析 content；没有就先调一次 `python3 scripts/chat_sync.py "看看我的整体歌曲表现"`（**此次仅用于内部提取 songId 映射，不展示给用户**）
   - 在 content 中精准匹配 `songName == "XXXX"` 的条目，取出对应 songId（假设为 `987654`）
2. 调用单曲分析：
   ```bash
   python3 scripts/chat_sync.py "《XXXX》这首歌最近表现怎么样" 987654
   ```
3. **完整原样**展示返回的 Markdown
4. 由于**已传 `songId`**，属于单首歌分析 → **不要**追加"末尾引导语"

### 场景三：配额用尽

**返回**：`code != 0`，message 类似"您今天在数据助手的使用次数达 10 次啦！"

**Agent 行为**：不要重试，直接告知用户："您今天的数据助手提问次数已达上限，欢迎明天再来~"

### 场景四：歌曲安全打击

**返回**：`code != 0`，message 为"歌曲不支持分析"

**Agent 行为**：告知用户："很抱歉，这首歌当前不支持数据分析。换一首试试？"；**不要**重试

### 场景五：超时

**返回**：`code == 0`，`data.finishReason == "TIMEOUT"`

**Agent 行为**：告知用户："数据分析耗时较长，请稍后再试~"；**不要**直接重试

### 场景六：用户指定单首歌，歌名与 content 中某首**相似但不完全相等** → 候选确认

> 本场景对应三档判定中的 **档 B**。**不得跳过用户确认直接发起单曲分析**。

**用户**：「帮我分析《文艺复兴》这首歌的表现」

**内部数据**：整体分析 content 中的单曲映射 `M` 里没有 `songName == "文艺复兴"` 的完全相等条目，但存在 `《文艺复兴(Remix)》`、`《文艺复兴 — 现场版》` 等明显相似条目；或出现多条 `《文艺复兴》` 同名异曲条目。

**Agent 行为（严格执行）**：

1. **不**调用 `chat_sync.py` 做单曲分析（因为 songId 尚未被用户确认）
2. **不**输出"匹配失败兜底话术"（那是档 C 专用）
3. **直接**向用户输出「候选确认话术」：

   ```
   没有找到与《文艺复兴》完全一样的歌曲，您是想分析以下哪一首呢？

   1. 《文艺复兴(Remix)》
   2. 《文艺复兴 — 现场版》

   请回复歌曲名或前面的编号；如果都不是，也可以告诉我"都不是"。
   ```

   - 候选歌名使用 `M` 中的原始 `songName`，外层用 `《》` 包裹
   - 候选数量最多 5 条
4. **等待用户回复**后再决定下一步：
   - 用户回复 `1` / `第二个` / 直接给出候选歌名 → 取出对应 songId，发起单曲分析（沿用用户最初的 userMessage 原文）
   - 用户回复"都不是" / 否定回答 / 无关回复 → 走场景七（兜底话术）
   - 用户给出全新歌名（不在候选内） → 当作新一轮提问，重新走 Step ①~④
5. **禁止**在候选确认话术中暴露任何技术字眼（`songId`、"映射"、"匹配"、"相似度"等）
6. **禁止**在候选确认话术中附带"播放量最高的那首是…"之类的引导性提示 —— 选择权完全交给用户

### 场景七：用户指定单首歌但歌名与 content **完全无关** → 固定兜底

> 本场景对应三档判定中的 **档 C**。**一定严格遵守，不得"变通"**。

**用户**：「帮我分析《一首根本不存在的歌》这首歌的表现」

**内部数据**：整体分析 content 中的单曲映射 `M` 里既没有完全相等条目、也没有任何明显相似条目；或者 `M` 本身为空（content 里完全没有单曲信息）；或者用户根本没给出明确歌名（如「这首」「那首歌」且上下文也无法定位）。

**Agent 行为（严格执行）**：

1. **不传** `songId`、**不退化**为"不传 songId 让服务端猜"、**不追问**用户补充信息
2. **不会**调用 `chat_sync.py`（即使调用也无法对应到用户想要的那首歌）
3. **直接**向用户输出固定话术（措辞不得修改）：

   ```
   很抱歉，暂时不支持对《一首根本不存在的歌》这首歌进行数据分析，您可以尝试分析其他歌曲~
   ```

   - 歌名使用用户原始表述（保留书名号/原外层写法）
   - 若用户未给出明确歌名（如「这首」「那首歌」），将 `《一首根本不存在的歌》` 整段替换为「该歌曲」
4. **禁止**在回复中出现任何技术字眼：`songId`、"映射"、"匹配失败"、"系统未识别到该歌曲"、"请提供 songId"、"请确认歌曲全名"等
5. **禁止**在此回复末尾追加「整体数据分析的末尾引导语」（那只在整体分析场景下适用）
6. （可选）可在兜底话术之后柔性附一句引导（不强制）：

   ```
   您也可以问问整体歌曲表现、听众画像等整体维度的数据哦~
   ```

## 常见问答

**Q：用户一次提了多个问题（如"看看我的播放量，再帮我分析一下听众画像"），怎么办？**

A：**一次调用只传一个 `userMessage`**。建议**串行调用两次**，第二次调用时告知用户"接下来为您分析第二个问题"。

**Q：`songId` 没拿到确切值该怎么办？**

A：按三档处理：

- **用户问的是整体数据**（没指向具体单首歌）：**不传** `songId`，直接走整体分析
- **用户明确指向单首歌 → 档 A 精准命中**：直接传命中条目的 songId
- **用户明确指向单首歌 → 档 B 相似命中**（包含/被包含/大小写标点差异/同名多条）：**使用「候选确认话术」问用户**，选中后再用对应 songId
- **用户明确指向单首歌 → 档 C 完全无关**：输出「匹配失败固定兜底话术」，**绝不**退化为不传 songId 让服务端猜（服务端在已指定单曲的语境下猜错，输出的是无关歌曲分析，用户感知极差）

**Q：能否重写 `userMessage` 让它更"规范"？**

A：**不能**。原样透传用户的原始提问，服务端会自行理解自然语言。任何加工都可能丢失用户真实意图。
